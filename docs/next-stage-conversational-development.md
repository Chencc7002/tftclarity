# TFTAgent 下一阶段开发文档：对话式数据助手

更新时间：2026-07-11

## 1. 阶段定位

下一阶段不再把 TFTAgent 当作一个缩小版数据筛选网站，而是把它改造成面向局内使用的对话式数据助手。

用户的核心操作只有一个：用自然语言提问。系统负责理解条件、补全缺省上下文、查询结构化数据、计算指标，并用简洁结论加数据证据回答。段位、天数、样本阈值、排序方式、装备范围等控件只作为高级设置和纠错入口，不能成为完成普通查询的前置步骤。

目标体验接近 ChatGPT：

```text
用户：大师以上，霞带什么装备最强？

助手：当前条件下，推荐红霸符 + 海妖之怒 + 海妖之怒。

依据：
- 前四率：...
- 登顶率：...
- 平均名次：...
- 样本：...
- 数据更新时间：...

本次采用条件：2 星霞 · 大师/宗师/王者 · 近 3 天 · 默认主流阵容 · 普通装备

用户：我已经有红霸符了，另外两件呢？

助手：继承上次条件后，推荐补齐双海妖……
```

本阶段不是让 LLM 代替查询和计算。LLM 可以参与自然语言结构化、澄清和结果措辞；英雄、装备、羁绊、样本、名次分布、排序及最终数字仍必须由本地受控代码和结构化数据决定。

## 2. 当前确认的问题

### 2.1 自然语言没有成为真正的主入口

当前界面仍要求用户理解并手动操作样本阈值、装备类型、排序、天数、段位和默认阵容策略。输入框中的部分条件不会覆盖设置，例如“大师及以上”目前不能由确定性规则解析，LLM 关闭时会继续使用面板里的长期偏好。

这与产品定位冲突。用户不应该为了查询装备而先学习 MetaTFT 的筛选逻辑。

### 2.2 缺少单装备排行

当前核心服务比较的是完整三件套，尚不能回答：

```text
霞哪个单件装备最厉害？
霞最适合优先做哪一件装备？
大师以上霞的单装备前四率排行。
```

如果用户只问“哪个装备最厉害”而没有英雄或适用范围，系统也缺少自然的澄清对话。

### 2.3 默认阵容补全会直接失败为“未补羁绊”

已确认的实际查询中，用户输入“霞在携带红霸符的前提下什么出装最强”，查询最终是：

```text
unit = TFT17_Xayah
starLevel = [2]
ownedItems = [TFT_Item_RapidFireCannon]
traitFilters = []
defaultContextStrategy = popular
```

“2 星”来自系统默认值。默认阵容策略确实执行了，但默认阵容选择器使用独立的固定门槛 `minClusterSamples = 100`；当所有包含霞的合格阵容都低于该门槛或被特殊玩法规则排除时，选择器直接返回：

```json
{
  "found": false,
  "traitFilters": [],
  "warning": "未找到稳定主流阵容，未补羁绊"
}
```

它没有选择“样本最高但低于稳定门槛”的候选作为低置信降级，也没有向用户发起澄清。界面的查询样本阈值和默认阵容样本门槛还是两套独立值，容易产生“设置了 50，为什么 71 场阵容仍不采用”的困惑。

此外，当前默认上下文可能把阵容记录中的整组羁绊全部作为 Explorer 过滤条件。下一阶段必须验证 MetaTFT 多羁绊参数的实际语义，避免把次要羁绊、唯一羁绊和阵容标签全部作为 AND 条件，导致查询被过度收窄。

### 2.4 当前 patch 的装备名称与可用性被旧规则覆盖

当前赛季复用了部分历史内部 ID，但本地仍按旧含义解释：

| 内部 ID | 当前官方名称 | 当前错误表现 |
|---|---|---|
| `TFT_Item_RapidFireCannon` | 红霸符 | 手工简称仍显示“火炮” |
| `TFT_Item_RunaansHurricane` | 海妖之怒 | 手工简称仍是“分裂弓”，并被标记为已移除 |
| `TFT_Item_MadredsBloodrazor` | 巨人杀手 | 手工简称仍显示“红叉” |

MetaTFT 原始响应已经返回：

```text
TFT_Item_RapidFireCannon
+ TFT_Item_RunaansHurricane
+ TFT_Item_RunaansHurricane
```

即当前的“红霸符 + 双海妖”。但永久的 `patch: current` 手工黑名单把 `TFT_Item_RunaansHurricane` 标记成：

```text
category = removed_or_legacy
current = false
obtainable = false
```

普通装备过滤器因此删除所有含海妖的组合，导致正确结果无法进入排名。这不是 LLM、样本门槛或用户输入问题，而是跨赛季 ID 复用与过期覆盖规则造成的数据层错误。

### 2.5 三件套“备选”容易被误解为锁定装备

当前推荐、备选 1、备选 2 只是排序前三的完整三件套，不保证差异性。如果三套都包含羊刀，用户会合理怀疑系统默认锁定了羊刀。

当前查询结构中 `ownedItems` 可能为空，但界面没有显著显示“已携带：无”，也没有解释“羊刀在前三套中重复出现是因为高样本组合共享核心装备”。结果展示缺少共同核心、候选差异及锁定状态的清晰区分。

### 2.6 查询条件、数据口径与回答证据没有形成一体化表达

当前页面能显示部分样本和指标，但不是完整的“结论—证据—条件—来源”回答。用户难以快速判断：

- 哪些条件来自本次输入；
- 哪些来自上一轮对话；
- 哪些是用户长期偏好；
- 哪些是系统默认或低置信补全；
- 当前结果来自实时数据、热缓存还是过期缓存；
- 某个结论为什么成立，以及有哪些风险。

## 3. 统一产品原则

### 3.1 条件优先级

所有查询统一采用以下优先级：

```text
本轮输入中的明确条件
> 当前对话中可安全继承的条件
> 用户长期偏好
> 系统默认值
```

本轮输入只覆盖明确提到的字段，不应意外清空其他可继承条件。发生冲突或歧义时，不猜测关键业务条件，应通过一轮简短对话澄清。

每个最终查询字段必须保留来源：

```json
{
  "rankFilter": {
    "value": ["MASTER", "GRANDMASTER", "CHALLENGER"],
    "source": "current_input",
    "confidence": 1
  }
}
```

前端可据此显示“用户指定”“沿用上轮”“偏好默认”“系统补全”等状态。

### 3.2 高频条件必须在 LLM 关闭时可用

以下条件需要确定性规则优先覆盖：

- 英雄、星级、装备数量；
- 已持有、排除、比较和指定装备；
- 普通、光明、神器、纹章及特殊装备范围；
- 段位及范围表达，例如“大师以上”“钻石到宗师”“只看王者”；
- 近 N 天、当前版本、样本至少 N；
- 前四优先、登顶优先、平均名次、样本最多；
- 完整三件套、单装备排行、剩余装备补齐和阵容榜意图。

LLM 用于处理更自由的口语、复杂省略、低置信实体和结果自然语言组织。所有模型输出必须经过严格 schema、本地实体目录和业务校验。

### 3.3 数据先于措辞

回答中的装备、阵容、样本、百分比、均名、更新时间和排名必须来自结构化查询结果。LLM 不得补齐缺失数字、猜测当前版本装备、复制第三方不可解释等级或把相关性描述成因果关系。

## 4. Patch-aware 装备目录重构

### 4.1 权威来源顺序

当前 patch 的官方装备目录应决定：

- canonical 中文名；
- 官方英文名；
- 当前赛季身份；
- 是否属于当前装备池；
- 基础类别和可获得性。

MetaTFT `/items` 用于确认统计接口实际出现的 API ID 和数据范围；Riot/Data Dragon 用于英文、图标及版本化资源回退。不同来源发生冲突时必须记录审计结果，不能静默采用历史手工值。

### 4.2 手工覆盖的边界

手工配置主要维护俗称、拼音、缩写和经审核的例外，不应永久覆盖当前官方 canonical 名称。

可用性黑名单必须绑定明确版本或赛季，例如：

```json
{
  "apiName": "TFT_Item_RunaansHurricane",
  "patch": "16.12",
  "season": "2026.S16",
  "current": false,
  "reason": "...",
  "source": "..."
}
```

禁止使用无法自然过期的永久 `patch: current` 规则。若确需 current 规则，必须带来源、生成时间、失效条件和发布审计，并在官方身份变化时让构建失败。

`shortName` 也必须 patch-aware。当前官方名称发生变化时，旧名称只能保留为别名，不能继续作为默认展示名。

### 4.3 必须修复的当前映射

- `TFT_Item_RapidFireCannon` 默认显示“红霸符”，“火炮”只保留为历史别名。
- `TFT_Item_RunaansHurricane` 默认显示“海妖之怒”，“分裂弓/飓风”只保留为历史别名；当前 patch 不得标记为不可用。
- `TFT_Item_MadredsBloodrazor` 默认显示“巨人杀手”，“红叉/麦瑞德”只保留为历史别名。
- “红霸符 + 双海妖”必须通过普通装备过滤并进入霞的候选排名。

### 4.4 测试迁移

现有“`TFT_Item_RunaansHurricane` 必须被过滤”的测试属于旧赛季事实，必须更新，不能继续锁定错误行为。

需要新增：

1. 当前 patch 官方目录覆盖历史手工简称的测试。
2. 同一 API ID 跨赛季复用的 fixture 和回归测试。
3. 手工可用性规则必须绑定明确 patch/season 的 schema 测试。
4. `patch: current` 永久黑名单审计失败测试。
5. “红霸符 + 双海妖”原始行经过目录合并、普通装备过滤、排序和 UI 序列化后仍存在的端到端测试。
6. 旧名称仍可作为输入别名，但输出使用当前官方名称的测试。
7. 每次装备目录刷新时，对 canonical 名称、类别、可用性和手工 override 漂移进行审计。

## 5. 对话式查询架构

### 5.1 统一意图

下一阶段至少支持：

```text
unit_build_rankings       单英雄完整三件套排行
unit_item_rankings        单英雄单装备聚合排行
unit_build_completion     已有一至两件装备时推荐补齐
unit_item_comparison      比较两个或多个装备
comp_rankings             全局阵容排行
clarification             缺少关键条件或歧义时追问
```

所有意图共享统一的约束结构，不再为每个页面维护一套独立筛选状态。

### 5.2 对话状态

对话状态应保存结构化查询，而不是拼接历史文本。每轮需要记录：

- 用户原始输入；
- 解析后的意图和条件；
- 每个条件的来源与置信度；
- 使用的数据版本、缓存和默认上下文；
- 返回的候选 ID；
- 允许下一轮继承的字段。

以下示例必须正确工作：

```text
用户：大师以上霞什么装备最强？
用户：我已经有红霸符。
用户：近一天呢？
用户：不要海妖，换一套。
```

后三轮应继承霞、段位及未被覆盖的条件，同时明确展示本轮发生的变化。

### 5.3 澄清策略

“哪个装备最厉害”没有英雄时，不应输出全英雄混合结论。助手应追问：

```text
想看哪个英雄的单装备表现？装备强度会随英雄和阵容变化。
```

若两个默认阵容样本接近但羁绊差异显著，应提供 2 至 3 个可点击候选，而不是静默选择或完全放弃上下文。

## 6. 单装备排行

### 6.1 统计定义

对于指定英雄和查询条件，将合法的完整装备组合按“是否包含某装备”聚合。对每件装备累加对应组合的八档 `placement_count`，计算：

```text
games
top4Rate
winRate
avgPlacement
buildCoverage / pickRate（仅在分母定义可靠时）
```

同一套组合中出现两件相同装备时，“该装备是否出现”的单件排行只计入一次该组合样本；另行提供 `copyCount = 1/2/3` 维度，避免把“双海妖”与“任意含海妖”混为一谈。

聚合前必须先完成：

- 当前 patch 装备身份与可用性校验；
- 装备策略过滤；
- 英雄、星级、羁绊、段位和时间条件过滤；
- 重复行及等价组合归一化。

### 6.2 结果解释

单装备排行表达的是“在当前条件下，包含该装备的完整出装样本表现”，属于条件相关性，不是装备单独造成结果的因果证明。

每个单装备结果至少显示：

- 装备名和图标；
- 前四率、登顶率、均名和样本；
- 样本覆盖；
- 最常见的 2 至 3 个搭配；
- 低样本或明显选择偏差提示。

## 7. 默认阵容与羁绊补全

### 7.1 选择与降级

用户未指定羁绊时：

1. 选择包含目标英雄且满足完整性规则的样本最高主流阵容。
2. 若有达到稳定门槛的候选，采用样本最高者。
3. 若没有稳定候选，但存在有效候选，采用样本最高者作为“低置信默认阵容”，并显示样本与警告。
4. 若多个候选样本接近且羁绊差异显著，发起澄清。
5. 只有完全没有包含该英雄的有效阵容时，才允许“未补羁绊”。

查询样本阈值和默认阵容稳定门槛必须使用统一配置模型，界面需分别说明用途，不能让两个同名“样本”参数静默冲突。

### 7.2 条件口径

如果 `/comps` 端点不支持 days/rank，回答必须说明默认阵容来源的实际口径，不能因为缓存 key 带有 days/rank 就声称阵容也经过了这些条件过滤。

需要验证传入 Explorer 的羁绊列表语义。默认阵容应提取对目标英雄和阵容形态有区分度的主要羁绊，不应未经验证把所有次要、唯一、职业和阵容标签同时作为强过滤条件。

### 7.3 默认值回显

当用户没有指定星级时，可以继续默认 2 星，但回答必须显示：

```text
2 星（系统默认）
```

同理，默认阵容、段位、天数、装备范围和样本门槛都要显示来源，避免用户把系统补全误认为自己输入的条件。

## 8. 三件套结果与候选多样性

默认回答应保留统计意义上的真实第一名，同时改善备选解释：

- 明确显示 `已携带：无` 或已锁定装备，不能只依靠颜色暗示。
- 若前三套共享某装备，显示“高频核心：羊刀（前三套均出现）”。
- 说明备选与推荐的差异装备及指标变化。
- 可提供“严格统计前三”和“差异化备选”两个视图。
- 差异化备选只能从真实候选中选择，不能为了视觉差异生成不存在的组合。
- 如果低样本组合指标更高，必须与稳定推荐分区显示，不能混成同一强结论。

## 9. ChatGPT 风格前端

### 9.1 主界面

前端改成消息流而不是固定查询表单：

- 顶部保留极简产品标识和状态入口；
- 中间为可滚动的用户/助手消息历史；
- 底部固定输入框和发送按钮；
- 支持 Enter 发送、Shift+Enter 换行；
- 查询中显示阶段状态，例如“理解条件”“查询数据”“计算排名”；
- 支持停止、重试、基于上一轮继续追问；
- 保留 Windows 小窗、置顶和全局热键能力。

不要求逐像素复制 ChatGPT，但应采用相同的信息结构：输入优先、对话连续、结论可读、证据可展开。

### 9.2 助手回答结构

每条数据回答按以下顺序组织：

1. 一句话结论。
2. 推荐装备或阵容卡。
3. 支撑结论的核心指标和样本。
4. 备选及差异解释。
5. 本次实际生效条件。
6. 数据来源、更新时间、缓存状态和风险。

“本次条件”和“完整数据来源”可以折叠，但结论、关键样本和低样本风险必须首屏可见。

### 9.3 条件控件

解析后的条件以可编辑标签显示，例如：

```text
[霞] [2 星·默认] [大师以上·用户指定] [近 3 天·偏好] [普通装备·默认]
```

用户可以点击标签修改，也可以直接继续用自然语言纠正。现有大块设置面板移入“高级设置”，只保留：

- 长期默认值；
- 数据与隐私状态；
- LLM 模式；
- 缓存和别名审核等开发/管理能力。

普通用户不需要看到别名审核、缓存类型或内部解析模式。

### 9.4 响应式要求

- 460px 为主要桌面小窗宽度，360px 必须可用。
- 输入框、消息、条件标签、装备卡和证据区不能横向溢出。
- 长段位、羁绊和数据来源允许换行。
- 加载、失败、低样本、过期缓存、澄清和空结果都要有独立视觉状态。
- 消息历史较长时仍要保持底部输入区可用。

## 10. 后端响应建议

统一返回对话消息与可验证数据块：

```json
{
  "conversationId": "...",
  "messageId": "...",
  "type": "unit_build_rankings",
  "answer": {
    "summary": "推荐红霸符 + 双海妖。",
    "evidence": [],
    "warnings": []
  },
  "query": {
    "intent": "unit_build_rankings",
    "constraints": {},
    "constraintSources": {},
    "defaultContext": {}
  },
  "results": [],
  "source": {
    "provider": "MetaTFT",
    "endpoint": "...",
    "updatedAt": "...",
    "cache": "live"
  },
  "clarification": null
}
```

`summary` 可以由受控模板或 LLM 生成，但必须仅引用 `results/evidence/query/source` 中存在的事实。

## 12. 验收用例

以下用例必须在 LLM 关闭时通过高频规则路径：

1. `大师以上霞什么三件装备最强？`
   - 段位解析为大师、宗师、王者；无需打开设置。
2. `霞在携带红霸符的前提下什么出装最强？`
   - 红霸符锁定；未指定羁绊时采用样本最高有效阵容。
3. `霞哪个单件装备表现最好？`
   - 返回单装备聚合排行及常见搭配。
4. `哪个装备最厉害？`
   - 追问目标英雄或范围，不输出无口径的全局结论。
5. 上一轮查询霞后输入 `近一天呢？`
   - 只覆盖 days，继承英雄及其他安全条件。
6. `不要海妖，换一套。`
   - 继承上下文并将海妖加入排除项。
7. 当前 patch 的 `TFT_Item_RunaansHurricane`
   - 输出“海妖之怒”，可作为普通装备；旧名称仍可解析。
8. 霞的原始“红霸符 + 双海妖”数据行
   - 不被 legacy 过滤，能按真实指标进入候选。
9. 没有达到默认阵容稳定门槛但存在低样本候选
   - 使用低置信默认阵容并显示警告，不直接显示“未补羁绊”。
10. 三个推荐都共享羊刀
    - 明示已携带为空、共同核心和备选差异，不造成默认锁定误解。
11. 数据源失败并使用过期缓存
    - 回答明确显示缓存时间和过期风险。
12. 任意回答中的百分比、样本和名称
    - 都能追溯到结构化结果和当前 patch 目录，LLM 无法注入不存在的数字。

## 13. 测试与发布门槛

- 单元测试覆盖解析、条件来源、会话合并、目录权威顺序、单装备聚合、默认阵容降级和响应 schema。
- 离线 fixture 覆盖当前装备 ID 复用、红霸符 + 双海妖、低样本默认阵容和多阵容歧义。
- 小窗 HTTP smoke 覆盖多轮对话，不只覆盖单次请求。
- 视觉 smoke 覆盖 360px/460px 的长对话、条件标签、单装备榜、完整三件套、澄清、加载、错误、低样本和 stale 状态。
- 实时 smoke 只作为人工发布证据，不进入默认离线测试。
- 发布前运行装备 patch 审计；发现官方名称/身份与手工 override 冲突时必须失败。
- 不得把未执行的 LLM、实时网络或视觉检查写成通过。

## 14. 本阶段完成定义

只有同时满足以下条件，下一阶段才算完成：

1. 用户无需操作筛选器即可通过一句话完成常用查询。
2. 连续追问能够可靠继承和覆盖条件。
3. 单装备排行、完整三件套、已有装备补齐和阵容榜均有明确意图与数据口径。
4. 当前 patch 官方装备目录决定默认名称和可用性，历史手工规则不会污染当前赛季。
5. 默认阵容能够稳定选择、低样本降级或发起澄清。
6. 回答包含结论、数据证据、生效条件、来源、更新时间和风险。
7. 前端形成 ChatGPT 风格的对话主界面，高级筛选不再阻断普通用户。
8. 所有关键路径有离线回归、HTTP smoke 和视觉证据。

## 15. 2026-07-12 实施结果

本阶段 P0-P3 已落地：

- 当前 patch 官方目录重新成为 canonical 名称、身份和可用性的权威来源；永久 Runaans 黑名单已移除，审计拒绝 `patch: current`/`*` 可用性规则。
- 查询统一为 `unit_build_rankings`、`unit_item_rankings`、`unit_build_completion`、`unit_item_comparison`、`comp_rankings` 和 `clarification`；条件带 `value/source/confidence`，会话按本轮、上轮、偏好、默认顺序合并。
- 单装备排行按装备存在性聚合八档名次，同一完整组合的重复件只累计一次样本，并输出 copyCount 和常见搭配。
- 默认阵容稳定候选不足但仍有有效候选时，会采用样本最高的低置信候选并警告；只有完全没有有效候选才不补羁绊。`/comps` 当前不支持 days/rank，来源说明不再宣称经过这些条件过滤。
- 小窗已替换为对话历史 + 固定输入框，支持连续追问、停止、重试、清会话、阶段状态、可编辑条件标签和数据来源/缓存风险。
